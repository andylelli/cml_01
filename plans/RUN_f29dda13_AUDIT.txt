E) DEEP DIVE SOLUTION PLAN (CONCRETE)
============================================================

Issue 1: Overall grade/score vs run outcome conflict
Problem surface:
- Score/grade indicates success while release gate determines failure/abort.

Likely root cause:
- Two different decision systems are both represented as top-level status without hierarchy.
- No canonical precedence field for final run outcome.

Concrete solution:
1) Introduce explicit top-level outcome contract in report schema:
- run_outcome: one of ["passed", "failed", "aborted"]
- run_outcome_reason: short machine-readable reason
- scoring_outcome: { score, grade, passed_threshold }
- release_gate_outcome: { status, hard_stop_count, warning_count }
2) Define deterministic precedence:
- If aborted=true or release hard-stop > 0, run_outcome = "aborted".
- Else if any required phase failed, run_outcome = "failed".
- Else run_outcome = "passed".
3) Keep existing score fields, but mark them informational, not authoritative.

Implementation notes:
- Update report generation in scoring aggregator and API export payload.
- Update UI labels: "Scoring Grade" vs "Run Outcome".

Acceptance criteria:
- A report can never show ambiguous top-level state.
- UI and export always display run_outcome first.
- Regression test: simulated run with A score + hard-stop must render run_outcome="aborted".

Issue 2: Validation issue count mismatch (11 critical vs 7 critical)
Problem surface:
- ValidationPipeline final count and release-gate count disagree with no stage attribution.

Likely root cause:
- Counts are computed from different snapshots (pre/post rewrite or filtered issue sets).
- No normalized issue lifecycle model.

Concrete solution:
1) Introduce issue_snapshot blocks by stage:
- validation_snapshots.pre_repair
- validation_snapshots.post_repair
- validation_snapshots.release_gate
2) Add issue lineage identifiers:
- issue_key = deterministic hash(type + chapter + entity + message template)
- Track resolved/remaining/new per stage.
3) Add consolidated summary:
- validation_reconciliation: { pre, post, release_gate, deltas }

Implementation notes:
- Extend release gate diagnostic details to include stage name and source dataset.
- Ensure export includes full reconciliation object.

Acceptance criteria:
- Every count shown in report can be traced to exactly one snapshot stage.
- A mismatch is explainable as stage delta, not unexplained disagreement.
- Test: synthetic run with repairs reduces 11 -> 7 and reconciliation reports delta=4 resolved.

Issue 3: Prose cost/duration mismatch between phase and diagnostic
Problem surface:
- phase totals and post_generation_summary totals differ materially.

Likely root cause:
- phase-level values include remediation/rewrite cycles, diagnostic values capture first generation pass only.
- Field names do not encode scope.

Concrete solution:
1) Replace ambiguous fields with scoped names:
- prose_duration_ms_first_pass
- prose_duration_ms_total
- prose_cost_first_pass
- prose_cost_total
2) Add generation_attempts metadata:
- rewrite_pass_count
- repair_pass_count
- token_usage_by_pass[]
3) Validate arithmetic invariants in report builder:
- total >= first_pass for duration and cost.

Implementation notes:
- Update both scoring logger diagnostic payload and report diagnostics projection.

Acceptance criteria:
- No ambiguous duration/cost fields remain.
- Automated schema test fails if any "prose_duration_ms" or "prose_cost" appears without scope suffix.

Issue 4: Clue visibility failed while NSD shows multiple clues revealed
Problem surface:
- Clue visibility metric says 0/1 while NSD trace indicates multiple clues surfaced.

Likely root cause:
- Metric and NSD trace use different definitions of "visible" (explicit canonical clue mention vs inferred reveal event).
- The evaluator may check stricter textual anchors than NSD update logic.

Concrete solution:
1) Define unified clue evidence model:
- clue_state: [introduced, hinted, explicit, confirmed]
- Each state transition must include chapter and text span evidence.
2) Require NSD reveal events to carry evidence references:
- evidence_quote
- evidence_offset (chapter, paragraph, sentence)
- confidence
3) Rework clue-visibility scorer to evaluate same clue_state model.
4) Add mismatch detector:
- If NSD reports reveal without textual evidence, raise telemetry warning "nsd_visibility_divergence".

Implementation notes:
- Add a strict parser for clue anchors in prose chapters.
- Keep backward compatibility by deriving old metric from new model during migration.

Acceptance criteria:
- For any run, clue visibility and NSD reveal counts are reconcilable by clue_state table.
- Test fixture where 7 clues are revealed must not score 0/1 unless evidence parsing fails (and then must emit divergence warning).

Issue 5: Transient report endpoint instability (404/500/200)
Problem surface:
- Polling clients observe unstable report states mid-run.

Likely root cause:
- Non-atomic report write/read lifecycle.
- Route attempts to parse incomplete file or race between writer and reader.

Concrete solution:
1) Atomic report writes:
- Write to temp file, fsync, rename to final target.
2) Stable in-progress response contract:
- Return 202 with { status: "in_progress" } when report not finalized.
- Avoid 404 for known run IDs once run starts.
3) Defensive read path:
- On parse failure of current report file, return last valid snapshot with stale=true.
4) Add endpoint-level circuit telemetry:
- report_read_state: [in_progress, partial, final, stale_fallback, error]

Implementation notes:
- Update activity logging scope for report polling outcomes.

Acceptance criteria:
- Under rapid polling, no 500 from report endpoint for active runs.
- 404 only for unknown run IDs.
- Load test: polling every 250ms during generation yields 202/200 only.

Issue 6: Prompt/style diversity drift and template leakage hard-stop
Problem surface:
- Overuse of similar opening styles and templated long-block duplication.

Likely root cause:
- Prompt constraints are advisory and not strongly penalized in-generation.
- No online repetition penalty feedback loop before commit.

Concrete solution:
1) Add pre-commit prose linter per chapter batch:
- opening-style entropy threshold
- n-gram duplication detector across chapter corpus
- repeated paragraph fingerprint detector
2) Add constrained decoding guardrail prompt block:
- prohibit reuse of previous opening pattern for N recent chapters
- require one of allowed varied rhetorical starts
3) If linter fails, force immediate local rewrite of chapter before acceptance.

Implementation notes:
- Persist linter metrics in diagnostics for postmortem.

Acceptance criteria:
- Reduced duplicatedLongParagraphCount over baseline runs.
- Opening-style entropy above configured threshold in >=95% passing runs.

Issue 7: Temporal consistency retries (month/season contradictions)
Problem surface:
- Chapter-level retries for month/season contradictions still leak into pipeline pressure.

Likely root cause:
- Temporal constraints are checked after generation rather than actively injected as hard constraints.

Concrete solution:
1) Build chapter-local temporal constraint packet from temporal context:
- canonical month
- allowed seasonal descriptors
- prohibited seasonal descriptors
2) Inject packet into prose generation prompt with explicit hard rule section.
3) Add deterministic preflight checker before chapter acceptance.
4) On violation, auto-rewrite only conflicting sentences, not full chapter.

Implementation notes:
- Keep a contradiction dictionary (month -> forbidden season terms) for fast checks.

Acceptance criteria:
- Month/season contradiction retries approach zero in sampled runs.
- Unit tests for month-to-season mapping and rewrite behavior pass.
