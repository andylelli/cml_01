PROSE QUALITY 100 MASTER PLAN
Scope: Agent 9 prose generation and all upstream/downstream components that affect prose chapter scoring.
Primary goal: Drive chapter-level scoring to 100/100 consistently, with no contradiction between events, diagnostics, release-gate state, and final report outcomes.
Owner surface: packages/prompts-llm/src/agent9-prose.ts + worker orchestrator + story-validation + scoring/report API.

====================================================================
0) SUCCESS DEFINITION (NON-NEGOTIABLE)
====================================================================
A. Chapter quality target
- Every generated chapter reaches 100/100 chapter-level score in internal scoring snapshots.
- Every chapter passes structure, continuity, scene grounding, clue visibility timing, and fair-play checks.

B. Run quality target
- Prose phase score: 100.
- Overall run score: 100.
- run_outcome must always align with run events and release-gate state.

C. Reliability target
- No stale/incomplete report served as final result.
- No missing post_generation_summary when prose generation has started.
- No template-linter contradiction (abort reason must match linter counters).

D. Fair-play target
- all_clues_visible = true
- discriminating_test_complete = true
- no_solution_spoilers = true
- no NSD clue reveal without evidence anchors

2) PRINCIPLE: 100/100 IS A SYSTEM PROPERTY, NOT A PROMPT PROPERTY
====================================================================
Treat prose quality as a multi-component contract:
- Inputs must be precise and complete.
- Prompt must be scoped, prioritized, and contradiction-free.
- Generation loop must enforce correctness before commit.
- Validators/scorers must measure what we actually care about.
- Telemetry must expose root causes in one run.
- Reporting must never lie about final state.

3) WORKSTREAM A - INPUT CONTRACT HARDENING (BEFORE PROMPT)
====================================================================
A1. Typed input normalization layer for Agent 9
- Build one normalization function that canonicalizes all optional inputs before prompt assembly.
- Remove all ad-hoc any usage at prompt time where feasible.
- Guarantee non-null defaults for required sections so prompt shape is stable.

A2. CML requirement extraction audit
- Validate that every prose requirement section maps to explicit chapter obligations.
- Produce a per-chapter requirement ledger before generation:
  - required clues
  - discriminating test requirements
  - suspect elimination obligations
  - culprit reveal obligations
  - setting anchors

A3. Outline-to-prose mapping integrity
- Add deterministic check that chapter index <-> scene mapping is complete and not shifted.
- Add chapter assignment checksum to detect accidental drift across retries.

A4. Clue distribution quality gate
- Fail early if clue metadata lacks concrete descriptive payload needed for prose surfacing.
- Ensure each clue has reader-observable anchor candidates before prose starts.

A5. Location profile usability gate
- Ensure each chapter has at least one resolvable location profile anchor.
- Detect chapters with generic/empty location fields and auto-inject deterministic fallback anchor.

A6. Temporal contract gate
- Validate temporal context has canonical month and derived season lock before prose.
- Build prohibited seasonal term list once and pass as deterministic chapter constraints.

A7. Cast integrity gate
- Enforce unique canonical names, known aliases, and pronoun map before generation.
- Add role-aware anonymous walk-on allowance list to reduce false name-invention errors.

A8. Locked facts quality
- Ensure locked facts are concise, conflict-free, and chapter-scoped where applicable.
- Reject locked facts that are ambiguous (example: vague quantities or unbounded times).

====================================================================
4) WORKSTREAM B - PROMPT ARCHITECTURE REWRITE
====================================================================
B1. Prioritized prompt sections
- Reorder system prompt by criticality:
  1) Hard constraints and fail conditions
  2) Fair-play and clue timing
  3) Continuity/setting/temporal locks
  4) Quality style guidance
  5) Output schema
- Remove duplicate or competing guidance lines.

B2. Constraint language simplification
- Replace long prose rules with atomic, testable directives.
- Convert high-risk rules into explicit must/must-not bullets.

B3. Token budget discipline
- Introduce prompt budget accounting for each context block:
  - continuity block
  - clue description block
  - location/temporal blocks
  - humour/craft guides
- Drop or summarize low-value sections when budget exceeds threshold.

B4. Batch-local requirement manifest
- Provide per-batch manifest table with chapter-by-chapter obligations.
- Include explicit yes/no completion checklist in prompt for each chapter.

B5. Anti-template prompt controls
- Add rolling opening-style exclusion list and forbidden phrase cache from prior chapters.
- Add chapter-specific lexical seeds tied to scene facts to force variation.

B6. Dialogue and voice controls
- Move character voice constraints into compact machine-readable mini-spec per character.
- Require at least one unique dialogue trait marker per speaking character in chapter.

B7. Deterministic fallback micro-prompts
- For retry attempts, generate compact error-specific corrective prompts rather than re-sending full prompt unchanged.
- Use category-specific repairs: scene_grounding, clue_visibility, temporal_consistency, etc.

====================================================================
5) WORKSTREAM C - GENERATION LOOP AND RETRY POLICY
====================================================================
C1. Retry strategy by error class
- Separate transient generation retries from validation-correction retries.
- Error-class policy:
  - structural JSON errors: immediate regeneration
  - continuity/setting errors: constrained rewrite
  - clue/test errors: mandatory evidence insertion rewrite
  - style entropy only: adaptive policy, never hide true logic errors

C2. Chapter rewrite granularity
- Add sentence-level surgical rewrite mode for narrow failures (temporal mismatch, anchor missing).
- Keep full-chapter rewrites only for multi-class failures.

C3. Entropy linter policy refinement
- Keep adaptive thresholds, but require correlation with repetitive content metrics before hard fail.
- Add confidence score to entropy warnings to reduce false positives.

C4. Mandatory pre-commit chapter checklist
- Before chapter commit, verify all required chapter obligations are met by deterministic checks.
- Do not commit chapter if obligations missing, even if prose sounds good.

C5. Stronger cross-batch learning
- Expand recurring error memory with last N failures and what fixed them.
- Inject concise negative examples from this run only (avoid prompt bloat).

C6. Chapter-level scoring feedback loop
- After each chapter, compute provisional component scores and feed deficits into next chapter constraints.
- Example: if clue visibility is weak in chapter N, chapter N+1 prompt adds stronger clue surfacing rules.

====================================================================
6) WORKSTREAM D - VALIDATION AND SCORING ALIGNMENT
====================================================================
D1. Single source of truth for clue visibility
- Ensure scoring adapter clue detection and NSD clue reveal use the same evidence model.
- Maintain clue states: introduced, hinted, explicit, confirmed.

D2. Evidence anchor completeness
- Require clue_evidence_anchors for every newly revealed clue in NSD trace.
- Add hard validation failure when revealed clue has no matching anchor.

D3. Discriminating test validator upgrade
- Validate all components:
  - setup
  - evidence usage
  - elimination logic
  - outcome declaration
- Map failures to exact chapter and paragraph.

D4. Scene grounding validator upgrade
- Enforce named location + sensory + atmosphere minimum in chapter opening block.
- Validate against location profile vocabulary and outline setting.

D5. Character consistency validator upgrade
- Detect illegal named walk-ons, alias drift, pronoun drift, and role drift.
- Distinguish harmless unnamed extras from forbidden named additions.

D6. Temporal consistency validator upgrade
- Add explicit month-season contradiction matrix and phrase normalizer.
- Validate both narrative text and dialogue references.

D7. Fair-play timing validator
- Ensure clue introduction and detective deduction are separated by chapter boundary when required.
- Fail same-chapter clue-discovery-plus-deduction patterns.

D8. Score rubric recalibration for 100/100 target
- Align rubric with desired output behaviors and remove ambiguous tests.
- Every failed test must include deterministic remediation instruction.

====================================================================
E1. Mandatory diagnostic payload completeness

E2. Event/report coherence gate

E4. Failure lineage bundle

  - error classes
  - corrective attempts
  - final blocking reason

E5. Prompt/response traceability
- Log prompt hash and context block sizes per chapter for debugging quality regressions.
- Avoid full sensitive prompt dumps; store structured fingerprints.

8) WORKSTREAM F - EVALUATION HARNESS AND BENCHMARKING
====================================================================
F1. Fixed-seed regression suite
- Create fixed seed run set with expected chapter-level outcomes.
- Include hard scenarios: high cast complexity, dense clues, strong temporal constraints.

F2. Chapter-level golden tests
- Snapshot expected chapter obligation fulfillment for each seed.
- Fail CI when chapter obligations regress.

F3. A/B prompt harness
- Test prompt variants with identical inputs and compare component scores.
- Keep winning variant only when statistically significant.

F4. Error taxonomy dashboard
- Track top failure classes over time:
  - clue visibility
  - scene grounding
  - discriminating test completeness
  - temporal contradictions
  - entropy/fingerprint/ngram issues

F5. Score decomposition monitoring
- Monitor validation, quality, completeness, consistency separately.
- Target 100 in all four components per chapter.

====================================================================
9) WORKSTREAM G - MODEL AND DECODING STRATEGY
====================================================================
G1. Decode parameter experiments
- Tune temperature/top_p by chapter phase (opening, mid, reveal).
- Use lower randomness for logic-critical scenes.

G2. Chapter type conditioning
- Tag chapter intent (setup, clue reveal, elimination, test, denouement).
- Use intent-conditioned sub-prompts and validator expectations.

G3. Two-pass generation for hard chapters
- Pass 1: logic skeleton and obligation satisfaction.
- Pass 2: style expansion preserving validated logic skeleton.

G4. Cost-quality governance
- Track cost per quality point and identify expensive low-yield retries.
- Apply targeted repair prompts before expensive full regeneration.

13) RISKS AND MITIGATIONS
====================================================================
Risk: Over-constrained prompts reduce prose naturalness.
Mitigation: Two-pass approach and selective style expansion after logic pass.

Risk: Token bloat from too many context blocks.
Mitigation: Hard budget and deterministic prioritization.

Risk: Validator strictness creates retry storms.
Mitigation: Error-class retries and surgical rewrites.

Risk: Scoring mismatch with desired prose quality.
Mitigation: Rubric calibration and benchmark-driven adjustments.

Risk: Hidden regressions in reporting truth.
Mitigation: Invariant checks + replay fixtures + API endpoint tests.

16) IMMEDIATE HOTFIX LANE (ONGOING ERROR RESPONSE)
====================================================================
Purpose
- Run a parallel fast-response track for current prose generation failures while phased architecture work proceeds.
- Reduce current failure rate in days, not sprints.

Trigger conditions
- Any rolling 10-run window where `agent9-prose` fails due to:
  - `Clue visibility` (critical), or
  - repeated `Word count target`/`All chapters present` completeness degradation.

Hotfix H1 - Clue visibility extraction parity (Priority 0)
Problem addressed
- Active critical completeness failures where scorer expects clues that adapter extraction does not reliably surface.

Actions
- Align expected clue ID source for scoring with the same clue contract used by prose prompting and NSD trace.
- Add dual-source clue ID diagnostics (`mapping_ids`, `distribution_ids`, `effective_ids`) to detect mismatch immediately.
- Add strict test fixtures where clues are semantically present but non-literal, and where mapping is partial.

Success metric
- `Clue visibility` critical failures reduced to near-zero on latest rolling 20 runs.

Hotfix H2 - Pre-commit completeness gate (Priority 0)
Problem addressed
- Chapters can still commit while trending toward low completeness score.

Actions
- Add deterministic pre-commit checks in generation loop for:
  - expected chapter count progress,
  - chapter minimum word threshold,
  - chapter obligation closure from requirement ledger.
- Block commit when checks fail and force targeted corrective rewrite.

Success metric
- Eliminate repeated `Word count target` major failures from first-pass outputs.

Hotfix H3 - Retry correction specificity (Priority 1)
Problem addressed
- Generic retries do not reliably fix completeness defects.

Actions
- Add micro-prompt templates per failure class (`clue_visibility`, `word_count`, `missing_chapter_obligations`).
- Include chapter-local required clue IDs and evidence insertion directives in retry context.

Success metric
- Reduced retries per failed batch and improved second-attempt correction rate.

Hotfix H4 - Score-path observability patch (Priority 1)
Problem addressed
- Slow triage when run fails but exact clue/obligation gap is not obvious.

Actions
- Persist per-run top missing clue IDs and missing chapter obligations in post-generation diagnostics.
- Emit scorer-to-adapter comparison snapshot for `expected vs extracted` clue sets.

Success metric
- Every failed run yields actionable root-cause fields without manual log archaeology.

Operational rule
- Hotfix lane merges must include tests and diagnostics fields in the same change.
- No hotfix may reduce invariant strictness for report truth.
