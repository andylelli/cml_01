# ANALYSIS 14 — Git-Tracked File Bloat: Audit and Ranked Deletion Plan

**Date:** 2026-03-28  
**Status:** Actionable — awaiting execution  
**Impact:** Reduces tracked file count from 8,827 → ~1,075 (88% reduction)

---

## 1. Problem Statement

Running `git ls-files | Measure-Object` against the `main` branch returns **8,827 tracked files**. The actual codebase (TypeScript source, config, docs, schema) is roughly **1,075 files**. The remaining ~7,750 files are build artefacts or package manager files that were committed before `.gitignore` was properly applied.

**Key rule**: once files are tracked by git, adding them to `.gitignore` has no effect. They remain tracked until explicitly unregistered with `git rm --cached`. No local files are deleted by this operation; it only removes them from version control.

---

## 2. Root Cause

At some point early in the repository's life, `node_modules/` and `dist/` directories were committed (likely via an initial `git add .` before the `.gitignore` was in place, or before it was correctly populated). The `.gitignore` already contains the correct rules:

```
dist/
node_modules/
logs/
```

…but those rules are silently ignored for already-tracked files.

## 4. Ranked Deletion Detail

### 4.1 — `node_modules/` (Priority: CRITICAL, 7,752 files)

The entire npm dependency tree — every package from `ajv` to `vue` — was committed. This accounts for **88% of all tracked files**. It includes source, dist, type definitions, and even nested `node_modules` inside packages like `@vue/compiler-core`.

**Root cause**: The root `node_modules/` was added to git. Additionally, workspace-level node_modules may also be tracked.

**Command to fix:**
```powershell
git rm --cached -r node_modules/
```

**Gitignore**: `node_modules/` is already present — no change needed.

**Risk**: Zero. `npm install` regenerates everything from `package-lock.json`.

---

### 4.2 — `dist/` in `apps/` and `packages/` (Priority: HIGH, 30 files)

Build output directories for three packages were committed:

| Path | Files |
|------|-------|
| `apps/api/dist/` | 12 |
| `apps/worker/dist/` | 12 |
| `packages/cml/dist/` | 6 |

Contains compiled `.js`, `.d.ts`, and `.js.map` files. Entirely regenerated by `npm run build`.

**Command to fix:**
```powershell
git rm --cached -r apps/api/dist/
git rm --cached -r apps/worker/dist/
git rm --cached -r packages/cml/dist/
```

**Gitignore**: `dist/` is already present — no change needed.

**Risk**: Zero. TypeScript compilation regenerates everything.

---

### 4.3 — `.tsbuildinfo` files outside `node_modules` (Priority: MEDIUM, 3 files)

| File |
|------|
| `tsconfig.base.tsbuildinfo` |
| `packages/prompts-llm/tsconfig.tsbuildinfo` |
| `packages/story-validation/tsconfig.tsbuildinfo` |

```

**Risk**: Zero. Deleted `.tsbuildinfo` files are recreated on next `tsc` run; first build will simply be non-incremental.

These are runtime quality report outputs produced by the scoring pipeline (e.g., `quality-report-run_2c17609e-fd30-45f5-9e34-1430bf9baa0c.json`). They are pipeline output artefacts, not source or configuration.

**Command to fix:**

**Gitignore addition needed:**

```

**Risk**: Low. These files exist locally and are not deleted. If you want to preserve run history, archive them separately before removing from git.

### 4.5 — `data/store.json.bak` (Priority: LOW-MEDIUM, 1 file, 2.1 MB)

A backup copy of the runtime database. No value in version control.

**Command to fix:**
```powershell

```

**Gitignore addition needed:**
```

```

**Risk**: Zero. The `.bak` file is not used by any code path.

---

### 4.6 — `data/store.json` (Priority: LOW — DECISION REQUIRED, 1 file, 4.4 MB)

The live runtime database containing all project state, run history, and generated content. Arguments:

| Keep in git | Remove from git |
|-------------|-----------------|
| Preserves run history across clones | 4.4 MB binary-ish JSON pollutes diffs |
| Useful for debugging specific runs | Not reproducible source; changes on every run |
| No other persistence layer | Should be excluded like all runtime data |

**Recommendation**: Remove from git tracking and add a `data/store.json.example` skeleton instead (empty `{ "projects": [], "runs": [] }`) so fresh clones can self-initialise.

**Command to fix (if decided):**
```powershell
git rm --cached data/store.json
```

**Gitignore addition needed:**
```
data/store.json
```

**Risk**: Medium — if removed, a fresh clone will need the app to create `store.json` on first run. Code should handle a missing store gracefully (likely already does).

---

### 4.7 — `stories/*.txt` (Priority: LOW, 1 file)

**Command to fix (if decided):**
```powershell

```

**Gitignore addition needed:**
```
stories/
```

**Risk**: Low. No code depends on these files.
