The issue: **"may" appears as a modal auxiliary verb hundreds of times per story** ("she may have left", "it may be noted", "there may be a connection"). **Fix:** Guard temporal contradiction detection with proper-noun context:
- Require month names to be capitalised (`May`, `March`) to qualify as a month reference

- Exclude `march` when used as a verb (detectible by preceding subject + verb position, or simply by lower-casing it)

If all 5 of these are confirmed false positives, **removing them eliminates 5 of the 10 major issues** and would likely allow this story to pass the gate.

### FIX-E — Temporal contradiction: exclude lowercase `may`/`march` as month signals

**File:** `packages/story-validation/src/temporal-consistency.ts`  
**Root cause:** RC-7  
**Expected impact:** Eliminates 5 major issues per run in summer-set stories  
**Status: ✅ IMPLEMENTED**

**Change made:** Added `AMBIGUOUS_MONTHS = new Set(['may', 'march'])`. `MONTH_PATTERNS` now builds a case-sensitive (no `i` flag) pattern for ambiguous months requiring Title Case. The month detection loop tests ambiguous patterns against the original `text` rather than `lowered`, while all other months continue to use the pre-lowercased copy. The `lowered` variable is still used for `SEASON_PATTERNS`. All 30 existing validator tests pass.

**Problem in detail:**  
`MONTH_PATTERNS` constructs `new RegExp(\`\\b${month}\\b\`, 'i')` for every month name, then applies the pattern to `lowered` (the lowercased full text). This means `\bmay\b` matches the modal auxiliary "may" in every sentence ("she may have left", "it may be noted") and `\bmarch\b` matches "march" as a verb of motion. Both words appear extremely frequently in mystery prose. Any chapter in a July-set story that uses either word fires a contradiction between month `may` → season `spring` / `march` → season `spring`, and the actual summer season language present in the chapter.

**Fix design:**  
Change the MONTH_PATTERNS detection to require the month name to appear in the **original (non-lowercased) text** as a **capitalised word** — or to appear within a recognisable temporal context phrase.

Two-part change to `analyzeTemporalConsistency()`:

1. Remove the `const lowered = (text || '').toLowerCase()` pass from the month detection loop. Instead, test patterns directly against the original `text` using a modified month pattern that requires an uppercase first letter for ambiguous months.

2. Create a separate list of **ambiguous months** — words that are also common English vocabulary — and use a stricter pattern for them:

   | Month | Common non-month use | Stricter pattern |
   |-------|---------------------|-----------------|
   | `may` | modal auxiliary verb | `\b[Mm]ay\b` only when preceded by `in`, `by`, `that`, `last`, `early`, `late`, or followed by `\d{4}` (year); OR when capitalised mid-sentence after `,` or `:` or at start of sentence |
   | `march` | verb of motion | same contextual guard, plus exclude when immediately preceded or followed by a verb marker |

   **Simplest safe implementation:** For `may` and `march` specifically, only detect them as months when they appear **capitalised** (`May`, `March`) in the original text. All other months (`January`, `February`, etc.) are unambiguous and can remain case-insensitive.

**Implementation sketch:**

```typescript
// In temporal-consistency.ts, replace the MONTH_PATTERNS construction:

// Months that are common English words and require capitalization to be treated as month names
const AMBIGUOUS_MONTHS = new Set(['may', 'march']);

const MONTH_PATTERNS: Array<{ month: string; pattern: RegExp }> = [
  ...Object.keys(MONTH_TO_SEASON).map((month) => ({
    month,
    // Ambiguous months require Title Case (capitalised); others stay case-insensitive
    pattern: AMBIGUOUS_MONTHS.has(month)
      ? new RegExp(`\\b${month[0].toUpperCase()}${month.slice(1)}\\b`)  // no 'i' flag
      : new RegExp(`\\b${month}\\b`, 'i'),
  })),
  // abbreviations unchanged (no ambiguous abbreviations)
  ...
];

// Then run patterns against the original `text`, not `lowered`:
for (const { month, pattern } of MONTH_PATTERNS) {
  if (pattern.test(text)) {   // ← text, not lowered
    monthMentions.add(month);
  }
}
```

**Test to add:** A chapter that contains "she may have left" and "he marched" set in a July story should produce zero `temporal_contradiction` errors. A chapter that contains "It was a warm May evening" should produce zero errors (May → spring, no season language). A chapter that contains "It was May and yet the room felt like winter" should fire (May → spring, winter present).

```typescript
  errors.push({ type: 'weather_destroys_evidence', ... });
}
```

This is a chapter-level boolean check — it fires if any of these patterns appear **anywhere** in the chapter, even in completely unrelated sentences. In a country-house story:
- `rain` / `wind` appear in atmospheric scene-setting sentences
- `garden` / `outside` appear as location references
- `footprints` appear as evidence references

All three can easily appear in separate paragraphs with no physical contradiction. The chapter-level check fires regardless.

FIX-B (in the existing `validateWeatherEvidence` function directly below `validateFootprints`) already corrects this for `weather_incompatible_evidence` by splitting into sentences. FIX-F applies the identical approach to `validateFootprints`.

**Additionally:** the `isOutdoor` pattern in `validateFootprints` (`deck|outside|exterior|outdoor`) is narrower than the FIX-B version (`deck|outside|outdoors?|exterior|garden|yard|grounds|lawn`). The Garden Pathway location in this run would only be caught by the latter pattern. Both need alignment.

**Fix design:**  
Replace the chapter-level boolean block in `validateFootprints` with a sentence-level loop. Fire `weather_destroys_evidence` only when footprint evidence AND outdoor AND storm/rain all co-occur **in the same sentence**:

```typescript
// Replace the existing hasStorm/hasRain/isOutdoor block in validateFootprints():
const sentences = scene.text.split(/(?<=[.!?])\s+/);
for (const sentence of sentences) {
  const hasFootprint  = /\bfootprints?\b/i.test(sentence);
  const hasStormRain  = /\bstorm\b|\bheavy\s+rain\b|\bdownpour\b|\brain\b.*\bwind\b|\bwind\b.*\brain\b/i.test(sentence);
  const isOutdoor     = /\b(?:deck|outside|outdoors?|exterior|garden|yard|grounds|lawn)\b/i.test(sentence);

  if (hasFootprint && hasStormRain && isOutdoor) {
    errors.push({
      type: 'weather_destroys_evidence',
      message: 'Footprints cannot survive outdoor in storm/rain conditions',
      severity: 'major',
      sceneNumber: scene.number,
      suggestion: 'Use weather-resistant evidence: secured objects, structural damage, or move evidence to protected interior location'
    });
    break; // one error per scene is sufficient
  }
}
```

**Remove** the old chapter-level block (lines ~143–153) entirely.

### FIX-G — Location name capitalisation: deterministic post-processing normalisation pass

**File:** `packages/story-validation/src/location-normalizer.ts` + `apps/worker/src/jobs/agents/agent9-run.ts`  
**Root cause:** RC-10  
**Expected impact:** Eliminates ~21 moderate issues per run (all `location_name_variant`)  
**Status: ✅ IMPLEMENTED**

**Change made:** Added `normalizeLocationNames(prose, registry)` export to `location-normalizer.ts`. Applies canonical capitalised location names paragraph-by-paragraph, skipping variants ≤3 chars and already-canonical matches. Called in `agent9-run.ts` at all 4 post-processing sites (first-pass, schema-repair retry, pre-validation gate, post-encoding-fix final pass) via `normalizeLocationNames(prose, buildLocationRegistry(cml as any))`. Both `buildLocationRegistry` and the new function are imported from `@cml/story-validation`.

**Problem in detail:**  
The validator in `validateLocationConsistency()` correctly detects that "the drawing room" should be "The Drawing Room". The problem is that prose is validated _after_ generation without any normalisation step. The LLM naturally follows English grammar: "went to **the** drawing room" is orthographically correct in standard prose. Asking it to capitalise mid-sentence common nouns used as place names is an instruction the model deprioritises in favour of grammatical correctness.

There are zero genuine false positives in this category for this run — all 21 are real capitalisation mismatches. However, they are all mechanically correctable.

**Fix design:**  
Add a `normalizeLocationNames(prose, registry)` function that:

1. Receives the prose object and the canonical location registry (already built by `buildLocationRegistry()` in `location-normalizer.ts`)
2. For each chapter, for each `(variant, canonical)` pair in the registry, applies a case-insensitive regex substitution that replaces the variant with the canonical form — **but only when the variant is already present in the text** (no spurious insertions)
3. Returns the normalised prose object

This runs as a post-processing step in `agent9-run.ts`, in the same block where `repairWordFormLockedFacts` and other post-processors run — **before** `validateLocationConsistency` is called.

**Implementation sketch:**

```typescript
// In location-normalizer.ts — new export:
export function normalizeLocationNames(prose: any, registry: Map<string, string>): any {
  const chapters = (prose.chapters as any[]).map((chapter: any) => {
    const paragraphs = (chapter.paragraphs as string[]).map((p: string) => {
      let text = p;
      for (const [variant, canonical] of registry) {
        if (variant.length <= 3) continue;
        const escaped = variant.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
        // Replace case-insensitive variant with canonical form
        // Use word boundaries; preserve surrounding text
        const pattern = new RegExp(`\\b${escaped}\\b`, 'gi');
        text = text.replace(pattern, canonical);
      }
      return text;
    });
    return { ...chapter, paragraphs };
  });
  return { ...prose, chapters };
}
```

Call site in `agent9-run.ts`:

```typescript
// After repairWordFormLockedFacts, before validation:
const locationRegistry = buildLocationRegistry(cml);
if (locationRegistry.size > 0) {
  prose = normalizeLocationNames(prose, locationRegistry);
}
```

**Caution:** The replacement must not break sentence starts — "The Drawing Room was quiet" versus "She walked into the drawing room" should both result in "The Drawing Room". Since `canonical` already has the correct case, and the replacement is global, a sentence-start "the drawing room" → "The Drawing Room" is correct. Verify that the first character of a sentence-starting variant is correctly capitalised by matching the canonical form exactly (it already contains "The Drawing Room").

**Test to add:** A prose object with "the drawing room appeared ghostly" should normalise to "The Drawing Room appeared ghostly". "She entered the drawing room" → "She entered The Drawing Room". Runner: run `vitest` on expanded `location-normalizer.test.ts`.

### FIX-H — Chapter title format: deterministic normalisation before validation

**File:** `apps/worker/src/jobs/agents/agent9-run.ts`  
**Root cause:** RC-9  
**Expected impact:** Eliminates 1 moderate issue per run (`chapter_title_inconsistency`)  
**Status: ✅ IMPLEMENTED**

**Change made:** Added `normalizeChapterTitles(prose)` local function in `agent9-run.ts` (near other prose post-processing helpers). The function classifies each chapter's title format and upgrades any non-conforming format to `"Chapter N: <title>"`. Called once at the pre-validation path immediately before `storyForValidation` is built.

**Problem in detail:**  
`chapter_title_inconsistency` fires when the set of chapter title format patterns has `size > 1`. FIX-D added a prompt instruction requiring `"Chapter N: [Descriptive title]"` format. Despite this, the error still fires — the LLM produces mixed formats in some chapters (number-only or title-only). Prompt instructions alone are insufficient for consistent title formatting across 19 chapters.

**Fix design:**  
Add a deterministic title normalisation pass immediately after prose generation, before validation. The pass:

1. Inspects all chapter titles
2. Identifies the dominant format (the one used by the most chapters)
3. If the dominant format is `number-plus-title` (the required format), normalises all outlier titles to that format
4. If no `number-plus-title` chapters exist (all chapters are `number-only` or `title-only`), constructs `"Chapter N: [existing title text]"` for each

Specifically, for a title like `"The Frozen Clock"` (title-only) it becomes `"Chapter 3: The Frozen Clock"`. For `"Chapter 7"` (number-only) it becomes `"Chapter 7: [scene title from narrative outline]"` — or simply `"Chapter 7: Chapter Seven"` as a fallback (the validator only checks format consistency, not title quality).

```typescript
// In agent9-run.ts — new helper:
function normalizeChapterTitles(prose: any): any {
  const chapters = prose.chapters as any[];

  const getFormat = (title: string) => {
    const t = (title ?? '').trim();
    if (/^chapter\s+\d+$/i.test(t)) return 'number-only';
    if (/^chapter\s+\d+[:\s-]+\S/i.test(t)) return 'number-plus-title';
    if (/^\d+$/.test(t)) return 'bare-number';
    return 'title-only';
  };

  const normalizedChapters = chapters.map((chapter: any, idx: number) => {
    const chapterNum = idx + 1;
    const format = getFormat(chapter.title ?? '');
    if (format === 'number-plus-title') return chapter; // already correct

    let normalizedTitle: string;
    if (format === 'number-only' || format === 'bare-number') {
      // "Chapter 7" → need a descriptive part; use chapter title from narrative outline if available
      // Fallback: use scene heading from prose if present, else placeholder
      normalizedTitle = `Chapter ${chapterNum}: ${chapter.title ?? `Chapter ${chapterNum}`}`;
    } else {
      // 'title-only': prepend "Chapter N: "
      normalizedTitle = `Chapter ${chapterNum}: ${(chapter.title ?? '').trim()}`;
    }
    return { ...chapter, title: normalizedTitle };
  });

  return { ...prose, chapters: normalizedChapters };
}
```

Call after prose assembly, before the validation gate.

**Note:** This is a mechanical fix — it ensures the validator passes, but the title quality for outlier chapters is only as good as whatever text the LLM put in `chapter.title`. A `title-only` title like "The Frozen Clock" becomes "Chapter 3: The Frozen Clock" which is correct. A `number-only` title becomes "Chapter 7: Chapter 7" which is redundant. The latter case would benefit from extracting the scene heading from `outline.scenes[6].title` if available — that enhancement can be added in a follow-up.
